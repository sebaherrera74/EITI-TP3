1 ceedling-dependencies
2 =====================
3 
4 Plugin for supporting release dependencies. It's rare for an embedded project to
5 be built completely free of other libraries and modules. Some of these may be
6 standard internal libraries. Some of these may be 3rd party libraries. In either
7 case, they become part of the project's ecosystem.
8 
9 This plugin is intended to make that relationship easier. It allows you to specify
10 a source for dependencies. If required, it will automatically grab the appropriate
11 version of that dependency.
12 
13 Most 3rd party libraries have a method of building already in place. While we'd
14 love to convert the world to a place where everything downloads with a test suite
15 in Ceedling, that's not likely to happen anytime soon. Until then, this plugin
16 will allow the developer to specify what calls Ceedling should make to oversee
17 the build process of those third party utilities. Are they using Make? CMake? A
18 custom series of scripts that only a mad scientist could possibly understand? No
19 matter. Ceedling has you covered. Just specify what should be called, and Ceedling
20 will make it happen whenever it notices that the output artifacts are missing.
21 
22 Output artifacts? Sure! Things like static and dynamic libraries, or folders
23 containing header files that might want to be included by your release project.
24 
25 So how does all this magic work?
26 
27 First, you need to add the `:dependencies` plugin to your list. Then, we'll add a new
28 section called :dependencies. There, you can list as many dependencies as you desire. Each
29 has a series of fields which help Ceedling to understand your needs. Many of them are
30 optional. If you don't need that feature, just don't include it! In the end, it'll look
31 something like this:
32 
33 ```
34 :dependencies:
35  :libraries:
36  - :name: WolfSSL
37  :source_path: third_party/wolfssl/source
38  :build_path: third_party/wolfssl/build
39  :artifact_path: third_party/wolfssl/install
40  :fetch:
41  :method: :zip
42  :source: \\shared_drive\third_party_libs\wolfssl\wolfssl-4.2.0.zip
43  :environment:
44  - CFLAGS+=-DWOLFSSL_DTLS_ALLOW_FUTURE
45  :build:
46  - "autoreconf -i"
47  - "./configure --enable-tls13 --enable-singlethreaded"
48  - make
49  - make install
50  :artifacts:
51  :static_libraries:
52  - lib/wolfssl.a
53  :dynamic_libraries:
54  - lib/wolfssl.so
55  :includes:
56  - include/**
57 ```

59 Let's take a deeper look at each of these features.
60 
61 The Starting Dash & Name
62 ------------------------
63 
64 Yes, that opening dash tells the dependencies plugin that the rest of these fields
65 belong to our first dependency. If we had a second dependency, we'd have another
66 dash, lined up with the first, and followed by all the fields indented again.
67 
68 By convention, we use the `:name` field as the first field for each tool. Ceedling
69 honestly doesn't care which order the fields are given... but as humans, it makes
70 it easier for us to see the name of each dependency with starting dash.
71 
72 The name field is only used to print progress while we're running Ceedling. You may
73 call the name of the field whatever you wish.
74 
75 Working Folders
76 ---------------
77 
78 The `:source_path` field allows us to specify where the source code for each of our
79 dependencies is stored. If fetching the dependency from elsewhere, it will be fetched
80 to this location. All commands to build this dependency will be executed from
81 this location (override this by specifying a `:build_path`). Finally, the output
82 artifacts will be referenced to this location (override this by specifying a `:artifact_path`)
83 
84 If unspecified, the `:source_path` will be `dependencies\dep_name` where `dep_name`
85 is the name specified in `:name` above (with special characters removed). It's best,
86 though, if you specify exactly where you want your dependencies to live.
87 
88 If the dependency is directly included in your project (you've specified `:none` as the
89 `:method` for fetching), then `:source_path` should be where your Ceedling can find the
90 source for your dependency in you repo.
91 
92 All artifacts are relative to the `:artifact_path` (which defaults to be the same as
93 `:source_path`)
94 
95 Fetching Dependencies
96 ---------------------
97 
98 The `:dependencies` plugin supports the ability to automatically fetch your dependencies
99 for you... using some common methods of fetching source. This section contains only a
100 couple of fields:
101 
102 - `:method` -- This is the method that this dependency is fetched.
103  - `:none` -- This tells Ceedling that the code is already included in the project.
104  - `:zip` -- This tells Ceedling that we want to unpack a zip file to our source path.
105  - `:git` -- This tells Ceedling that we want to clone a git repo to our source path.
106  - `:svn` -- This tells Ceedling that we want to checkout a subversion repo to our source path.
107  - `:custom` -- This tells Ceedling that we want to use a custom command or commands to fetch the code.
108 - `:source` -- This is the path or url to fetch code when using the zip or git method.
109 - `:tag`/`:branch` -- This is the specific tag or branch that you wish to retrieve (git only. optional).
110 - `:hash` -- This is the specific SHA1 hash you want to fetch (git only. optional, requires a deep clone).
111 - `:revision` -- This is the specific revision you want to fetch (svn only. optional).
112 - `:executable` -- This is a list of commands to execute when using the `:custom` method

115 Environment Variables
116 ---------------------
117 
118 Many build systems support customization through environment variables. By specifying
119 an array of environment variables, Ceedling will customize the shell environment before
120 calling the build process.
121 
122 Environment variables may be specified in three ways. Let's look at one of each:
123 
124 ```
125  :environment:
126  - ARCHITECTURE=ARM9
127  - CFLAGS+=-DADD_AWESOMENESS
128  - CFLAGS-=-DWASTE
129 ```
130 
131 In the first example, you see the most straightforward method. The environment variable
132 `ARCHITECTURE` is set to the value `ARM9`. That's it. Simple.
133 
134 The next two options modify an existing symbol. In the first one, we use `+=`, which tells
135 Ceedling to add the define `ADD_AWESOMENESS` to the environment variable `CFLAGS`. The second
136 tells Ceedling to remove the define `WASTE` from the same environment variable.
137 
138 There are a couple of things to note here.
139 
140 First, when adding to a variable, Ceedling has no way of knowing
141 what delimiter you are expecting. In this example you can see we manually added some whitespace.
142 If we had been modifying `PATH` instead, we might have had to use a `:` on a unux or `;` on
143 Windows.
144 
145 Second, removing an argument will have no effect on the argument if that argument isn't found
146 precisely. It's case sensitive and the entire string must match. If symbol doesn't already exist,
147 it WILL after executing this command... however it will be assigned to nothing.

149 Building Dependencies
150 ---------------------
151 
152 The heart of the `:dependencies` plugin is the ability for you, the developer, to specify the
153 build process for each of your dependencies. You will need to have any required tools installed
154 before using this feature.
155 
156 The steps are specified as an array of strings. Ceedling will execute those steps in the order
157 specified, moving from step to step unless an error is encountered. By the end of the process,
158 the artifacts should have been created by your process... otherwise an error will be produced.
159 
160 Artifacts
161 ---------
162 
163 These are the outputs of the build process. There are there types of artifacts. Any dependency
164 may have none or some of these. Calling out these files tells Ceedling that they are important.
165 Your dependency's build process may produce many other files... but these are the files that
166 Ceedling understands it needs to act on.
167 
168 ### `static_libraries`
169 
170 Specifying one or more static libraries will tell Ceedling where it should find static libraries
171 output by your build process. These libraries are automatically added to the list of dependencies
172 and will be linked with the rest of your code to produce the final release.
173 
174 If any of these libraries don't exist, Ceedling will trigger your build process in order for it
175 to produce them.
176 
177 ### `dynamic_libraries`
178 
179 Specifying one or more dynamic libraries will tell Ceedling where it should find dynamic libraries
180 output by your build process. These libraries are automatically copied to the same folder as your
181 final release binary.
182 
183 If any of these libraries don't exist, Ceedling will trigger your build process in order for it
184 to produce them.
185 
186 ### `includes`
187 
188 Often when libraries are built, the same process will output a collection of includes so that
189 your release code knows how to interact with that library. It's the public API for that library.
190 By specifying the directories that will contain these includes (don't specify the files themselves,
191 Ceedling only needs the directories), Ceedling is able to automatically add these to its internal
192 include list. This allows these files to be used while building your release code, as well we making
193 them mockable during unit testing.
194 
195 ### `source`
196 
197 It's possible that your external dependency will just produce additional C files as its output.
198 In this case, Ceedling is able to automatically add these to its internal source list. This allows
199 these files to be used while building your release code.

201 Tasks
202 -----
203 
204 Once configured correctly, the `:dependencies` plugin should integrate seamlessly into your
205 workflow and you shouldn't have to think about it. In the real world, that doesn't always happen.
206 Here are a number of tasks that are added or modified by this plugin.
207 
208 ### `ceedling dependencies:clean`
209 
210 This can be issued in order to completely remove the dependency from its source path. On the
211 next build, it will be refetched and rebuilt from scratch. This can also apply to a particular
212 dependency. For example, by specifying `dependencies:clean:DepName`.
213 
214 ### `ceedling dependencies:fetch`
215 
216 This can be issued in order to fetch each dependency from its origin. This will have no effect on
217 dependencies that don't have fetch instructions specified. This can also apply to a particular
218 dependency. For example, by specifying `dependencies:fetch:DepName`.
219 
220 ### `ceedling dependencies:make`
221 
222 This will force the dependencies to all build. This should happen automatically when a release
223 has been triggered... but if you're just getting your dependency configured at this moment, you
224 may want to just use this feature instead. A single dependency can also be built by specifying its
225 name, like `dependencies:make:MyTunaBoat`.
226 
227 ### `ceedling dependencies:deploy`
228 
229 This will force any dynamic libraries produced by your dependencies to be copied to your release
230 build directory... just in case you clobbered them.
231 
232 ### `paths:include`
233 
234 Maybe you want to verify that all the include paths are correct. If you query Ceedling with this
235 request, it will list all the header file paths that it's found, including those produced by
236 dependencies.
237 
238 ### `files:include`
239 
240 Maybe you want to take that query further and actually get a list of ALL the header files
241 Ceedling has found, including those belonging to your dependencies.

243 Testing
244 =======
245 
246 Hopefully all your dependencies are fully tested... but we can't always depend on that.
247 In the event that they are tested with Ceedling, you'll probably want to consider using
248 the `:subprojects` plugin instead of this one. The purpose of this plugin is to pull in
249 third party code for release... and to provide a mockable interface for Ceedling to use
250 during its tests of other modules.
251 
252 If that's what you're after... you've found the right plugin!
